## Why this exists

The training plan is the core product. The runner gets it essentially once, with
adjustments after. Two beliefs drive this work:

1. A plan the runner helped shape is one they commit to (IKEA effect). Involving
   them also surfaces constraints a one-shot generation misses (can't run Tuesdays,
   long run must be Saturday, hates the track).
2. The first plan is the highest-leverage single LLM call in the product. It is the
   first impression and it decides whether the runner stays. It is created once per
   user, so its cost amortizes over the whole relationship.

## Premise resolution — sequencing is the whole game

Co-creation must be **draft-first, not interview-first.** F2-b deliberately collapsed
multi-turn onboarding into one message to kill drop-off. A blank-page interview before
the runner sees any plan would reintroduce exactly that friction. So MARP builds a
strong draft fast, then opens specific doors for collaboration on the real artifact.
The IKEA effect works on editing something real, not on building from nothing.

## Locked decisions

- **D1 — Draft-first co-creation.** MARP generates the plan, presents it with 2-3
  targeted "I wasn't sure about X" hooks (long-run day, training-day fit, intensity
  comfort), runner edits via chat.
- **D2 — Model tiering, eval-gated.** Add a dedicated `planModel` config var. Run the
  plan eval suite (V10) on Sonnet vs Opus. If the quality delta is real, first-plan
  creation runs on Opus; adjustments stay on Sonnet. If the delta is marginal, stay on
  Sonnet and keep the spend. Decision is measured, not assumed.

## Accepted expansions (in v1.3 scope)

- **E1 — Plan revision history.** Snapshot each plan version; runner can "revert" /
  undo. The safety net that makes co-creation safe to attempt. (S-M, Low)
- **E3 — Regenerate-from-scratch.** "Make it 5 days", "more aggressive" → clean re-roll
  on a new constraint, distinct from incremental edits. (S, Low)
- **E4 — "Why this session?" explainer.** Surfaces the per-session `reasoning` we
  already store, on demand. Plan defends its own choices. (S, Low)
- **E5 — Two-option first draft.** MARP opens with two philosophies (e.g. conservative
  4-day vs aggressive 5-day); runner picks one, then collaborates on it. Deepest buy-in
  is choosing between real options. (M, Med — 2x first-plan cost on Opus)

## Build-order dependencies / resolutions

- **E5 reshapes D1.** Combined flow: generate two plans → runner picks one → draft-first
  hooks + edits on the chosen plan. "Draft-first" becomes "two-drafts-first." Build E5's
  presentation on top of D1's hooks, not separately.
- **D2 eval must cover E5.** The Sonnet-vs-Opus comparison has to evaluate two-option
  generation quality and cost, since that is the shipping flow, not single-draft.
- **E1 underpins E3/E5.** Revision history should snapshot before a regenerate (E3) or
  an option-switch (E5) so "go back" works across all edit types.
- **Inputs gate quality, not just the model.** Opus amplifies good inputs; it cannot
  personalize around constraints never captured. The draft-first hooks (D1) are how we
  capture the missing constraints. Ship D1's input-richness, then let Opus amplify it.

## Eng review — phasing (locked)

v1.3 is a subsystem, not one feature. The plan-mutation foundation does NOT exist today:
after the pivot saves the plan, it is a frozen artifact (no edit/adjust/regenerate path
anywhere in `process-incoming.ts`). D1/D2/E1/E3/E5 all presuppose editing. So we phase it,
each phase its own PR:

- **Phase 1 (this PR) — mutation foundation + Opus routing + verify E4.** Detect plan-edit
  intent, apply on Sonnet, re-save, present. Add `planModel`, route first-creation to it
  (eval-gated Opus). Emit draft-first hooks. Confirm E4 already works.
- **Phase 2 — E1 revision history + E3 regenerate.** Both layer on Phase 1's mutation.
- **Phase 3 — E5 two-option first draft.** Reshapes the pivot; biggest piece.

## Phase 1 architecture (eng review resolutions)

- **A1 — Edit application: targeted mutation.** New `adjustPlan(athleteId, editRequest)`:
  load stored plan → one Sonnet call (`domainModel`) with the current plan + the change →
  `parsePlan` validates → `saveAthletePlan`. Preserves the untouched structure; no full
  regen. Risk: model rewrites more than asked — mitigated by prompt discipline now, and
  E1 revert (Phase 2) makes an over-eager edit cheap to undo.
- **A2 — Intent detection: extend the existing classifier.** Add a `plan_edit` flag to the
  router classifier (already runs every message, already emits domains/complexity/is_fork).
  No new call, no added latency. `process-incoming` routes `plan_edit` → `adjustPlan` once
  the runner is past onboarding/pivot.
- **A3 — Draft-first hooks: in-plan field.** The generator emits `open_questions: string[]`
  (2-3 items) in the same call. Rendered after the plan summary. The runner's reply is a
  normal `plan_edit`, so hooks and edits share one path.
- **D2 routing — first-creation vs adjustment.** `generatePlan` switches model to
  `config.llm.planModel` (new var, default Opus pending the eval). `adjustPlan` uses
  `domainModel` (Sonnet). Two named knobs, explicit.
- **E4 — verify only.** `memory/retrieve.ts:288` already serializes the full plan (with
  per-session `reasoning`) into every coaching turn. "Why Tuesday intervals?" already
  reaches an expert with the reasoning in context. Confirm with a live test; add a prompt
  nudge only if it underperforms. No new code path planned.

### Data flow — the edit path

```
runner: "move my long run to Saturday"
  │
  ▼
classifier (1 Haiku call, existing)  ──►  { domains, complexity, plan_edit: true }
  │                                              │
  │ plan_edit === true AND onboarded             │ else → existing expert router
  ▼
adjustPlan(athleteId, editRequest)
  │  load stored plan (getStoredPlan)
  │  Sonnet call: current plan + edit  ──►  modified plan JSON
  │  parsePlan (throws on malformed → friendly retry, bounded 1x)
  │  saveAthletePlan (same write path as generate/ingest)
  ▼
reply: renderPlanSummary(modified)  +  "changed: long run → Saturday"
```

### Model routing

```
FIRST creation   generatePlan   → config.llm.planModel    (Opus, eval-gated)
ADJUSTMENT       adjustPlan     → config.llm.domainModel  (Sonnet)
```

## Test plan (Phase 1)

- **adjustPlan (unit):** happy path (valid edit → modified plan saved); malformed LLM
  output → one retry → friendly failure (no silent drop); edit with no stored plan →
  guard reply ("build a plan first"); edit that the model returns unchanged → still valid.
- **classifier (unit):** `plan_edit` parses true on edit phrasings ("move my Tuesday",
  "I can't run Wednesdays", "make week 3 easier"); defaults false on general coaching and
  when the field is absent (conservative, like `complexity`).
- **process-incoming (integration):** plan_edit routes to adjustPlan only when onboarded +
  past pivot; a plan_edit before any plan exists falls through to a guard, not a crash.
- **[→EVAL]** generator: Sonnet vs Opus on the V10 fixtures + `open_questions` quality.
  Define the "delta is real" bar BEFORE the run (e.g. Opus wins on ≥3/5 fixtures on the
  existing scoring, or a clear reasoning-quality gap). If marginal, stay Sonnet.

## Failure modes (Phase 1)

| Codepath | Realistic failure | Test? | Error handling | User sees |
|----------|-------------------|-------|----------------|-----------|
| adjustPlan | LLM returns malformed/truncated plan JSON | yes | parsePlan throw → 1 retry → friendly msg | clear "couldn't apply that, try rephrasing" |
| adjustPlan | model rewrites the whole plan, not just the ask | partial (eval) | prompt discipline; E1 revert (Phase 2) | over-broad change until revert ships |
| classifier | false-positive plan_edit on a coaching question | yes | conservative default; adjustPlan no-op-saves an unchanged plan | mild: a coaching Q treated as an edit |
| adjustPlan | edit arrives before a plan exists | yes | guard: "let's build your plan first" | clear guidance, no crash |

No silent-failure critical gaps: every path returns a runner-visible message.
