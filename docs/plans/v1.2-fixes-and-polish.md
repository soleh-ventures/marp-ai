# v1.2 — Onboarding fixes + conversational polish

Source: live testing notes from the founder (2026-06-05), 8 items. Mix of
bugs (root-caused below) and UX/feature changes. Branch target: `master`.

## Premise

v1.1 shipped the plan-anchored flow. Real use surfaced friction in the
*first ten minutes* — the part that decides whether a runner ever gets to
a plan. Three hard bugs, three UX upgrades, two architecture calls. Fixing
the first-ten-minutes funnel is higher leverage than any new feature.

---

## The 8 items, root-caused + scoped

### F1 — Strava connects, then silence (BUG)

**Symptom**: After OAuth, runner gets "✅ Strava connected! pulled N
activities" and the browser says "MARP will take it from here" — then
nothing. Runner is confused.

**Root cause**: `src/routes/strava-auth.ts:115` sends the connect
confirmation from the OAuth *callback* (browser → server, out-of-band). It
is terminal. Onboarding only starts inside `process-incoming.ts` on the
next **inbound** WhatsApp message (`!isOnboarded` branch, line 297). Nothing
bridges the two — the runner has to guess they should text again.

**Fix**: After backfill completes, append an onboarding kickoff to the
connect confirmation (one message: "Connected + N activities. Now, what
should I call you and what race are you chasing?"). The runner's reply then
lands in the onboarding branch as normal. Static kickoff copy — no extra
LLM call. Blast radius: `strava-auth.ts` + the onboarding prompt (so the
LLM knows Strava is already connected when the first answer arrives).

### F2 — Onboarding progress + "Why I ask" broken; merge into one message? (BUG + DESIGN)

**Symptom**: Progress line ("2 of 6") doesn't change across different
questions, and sometimes regresses (3 → 2). "Why I ask" likewise stale.

**Root cause**: `onboarding-stages.ts:69` derives both lines from
`nextSection`, which is `parsed.next_section` — **the LLM picks it**
(`onboarding.ts:157`). The LLM asks several questions inside one section
(so the tail is identical across turns) and can name an earlier section
(so progress regresses). Nothing enforces monotonic progression; progress
is section-granular, not question-granular.

**Two ways forward (TASTE — the founder asked to brainstorm):**

- **F2-a — Fix the multi-turn flow.** Make progress code-owned and
  monotonic: track a step counter in `OnboardingMeta`, advance it once per
  turn, never let it regress, derive the tail from that. Keeps the
  conversational feel. ~½ day CC.
- **F2-b — Single compact message (founder's idea).** Replace the N-turn
  Q&A with ONE structured message listing all fields, so the runner answers
  everything in one reply, free-form. One LLM extraction pass parses the
  reply; a short follow-up only for whatever's missing. Kills the progress
  bug entirely (no stages), cuts time-to-plan, fewer LLM calls. Risk: a
  wall-of-questions can feel like a form; needs tight copy. ~1 day CC.

Both subsume F3 (below). Recommendation surfaced at the gate.

### F3 — Stop asking fitness questions when Strava is connected (UX)

**Symptom**: Onboarding still asks "how many km/week" and "longest recent
run" even when Strava is connected and those are derivable.

**Root cause**: `runOnboardingTurn` (`onboarding.ts:81`) builds its payload
from a thin slice — it never passes Strava status or recent-activity
summary to the LLM, and the onboarding prompt has no rule to skip fitness
questions when the data already exists.

**Fix**: Inject a recent-activity summary (weekly volume, longest run,
derived from `activities`) into the onboarding payload when Strava is
connected, and add a prompt rule: "if fitness data is present, confirm it
in one line instead of asking." Couples with whichever F2 path wins.

### F4 — Smarter model tiering + answer length (ARCHITECTURE)

**Symptom**: Replies are often long; founder worried about token cost.

**Current state**: Every onboarded free-form message runs the full router:
classifier (Haiku) → domain (Sonnet) → synthesizer (Sonnet). Even "thanks!"
pays for two Sonnet calls. Synthesizer caps at 250 words but that's still
long for small talk.

**Decision (TASTE — close approaches):** how to decide cheap-and-short vs
deep-and-long.
- **F4-a — Extend the existing classifier** to emit a `complexity` signal
  (small_talk | simple | coaching). small_talk/simple → single Haiku call,
  ≤40 words, no synthesizer. coaching → current Sonnet pipeline. Reuses the
  classifier we already pay for (DRY). 
- **F4-b — Cheap pre-filter heuristic** (length/keyword) before the
  classifier. Faster, but brittle and duplicates intent the classifier
  already has.
Recommendation surfaced at the gate. Either way: add an explicit
length contract per tier so Haiku answers stay friend-short.

### F5 — Typing indicator instead of "thinking…" (RESEARCH-GATED)

**Symptom**: Founder wants the "…is typing" affordance, not a literal
"thinking…" message.

**Reality**: WhatsApp Business added typing indicators in 2025, but
delivery depends on **Twilio exposing it** on the WhatsApp channel. Unknown
today. This is a research spike: confirm Twilio API support; if yes, swap
`fireThinkingAck` to emit the indicator; if no, keep "thinking…" (it's a
reversible one-liner). Don't build blind.

### F6 — Show the plan's methodology up front (UX)

**Symptom**: The plan is personalized and cites per-session reasoning, but
the *summary* (`renderPlanSummary`, `types.ts:208`) doesn't state the
overall method, so the runner can't see at a glance that it's legit.

**Fix**: Generator emits a top-level `methodology` string (e.g. "Built on:
Pfitz-style base→build→peak→taper, 80/20 polarized intensity, 10%-rule
weekly progression"). `renderPlanSummary` shows it as the first line. Prompt
already constrains to recognised principles, so this is low-risk. Small.

### F7 — Ask reminder timing: night-before vs morning-of (FEATURE)

**Symptom**: Reminders only fire morning-of. Founder wants the runner to
choose night-before (evening) or morning-of.

**Current state**: `reminder_prefs = { enabled, time_local }`; scheduler
sends on the session day. **Fix**: add `timing: "morning_of" |
"night_before"` to prefs; capture it in the reminder prompt; scheduler
sends night-before reminders on the evening preceding a training day.
Extends V8 cleanly (prefs schema + scheduler + capture classifier).

### F8 — Date/weekday wrong ("5 June = Thursday", should be Friday) (BUG)

**Root cause**: Two compounding issues.
1. `buildUserPayload` (`onboarding.ts:213`) and the plan generator
   (`generator.ts:26`) inject only `new Date().toISOString().slice(0,10)` —
   a bare **date string with no weekday**. When the LLM states a weekday it
   computes it itself, and LLMs are unreliable at date→weekday.
2. `toISOString()` is **UTC**. Late evening in Berlin (UTC+2) is already the
   next day in UTC, so "today" can be off by one for the runner.

(The reminder scheduler is correct — it uses `Intl.DateTimeFormat` with the
athlete's timezone. Verified Friday in the smoke test.)

**Fix**: Compute today's date **and weekday** in the athlete's timezone via
`Intl.DateTimeFormat` and inject both into every LLM payload that reasons
about dates (onboarding + generator). Never let the LLM derive the weekday.

## Locked decisions (/autoplan gate, 2026-06-05)

- **F2 → F2-b**: single compact onboarding message + ONE gap-fill
  follow-up. Deletes the progress-bar bug class entirely (no stages).
  Must re-ask injury if the runner's single reply omits it (safety).
- **F4 → F4-a**: extend the existing Haiku classifier with a
  `complexity` signal (small_talk | simple | coaching). **Conservative
  escalation**: only drop to Haiku on high-confidence small talk; default
  to Sonnet when unsure. Mis-routing a coaching question to Haiku is the
  expensive failure, not an over-spent ack.
- **F5 → spike first**: 1–2 hr research on Twilio WhatsApp typing-
  indicator support, then decide. "thinking…" stays as fallback.

## Review findings folded in (under-scoped in first draft)

1. **F8**: timezone is NULL during onboarding (set only at plan gen) →
   use `inferTimezoneFromPhone()` as the onboarding-time source. Compute
   `start_date` (week-1 Monday) in code in the generator, not in the LLM.
   Add one shared `nowInZone()` helper used at every date-injecting site.
2. **F1**: guard against double onboarding-kickoff — only kick off if
   onboarding hasn't already started (runner may connect Strava AND text).
3. **F7**: night-before is a separate scheduler query (tomorrow's session,
   evening window) with Sat-night→Sunday-long-run + DST edges. Bigger than
   a prefs field.
4. **F4**: apply a shorter length contract to the Sonnet synthesizer too
   (~120 words default), not only the Haiku tier.
5. **Tests**: F2-b invalidates V3 progress tests + onboarding tests; add
   single-reply extraction tests. Consider an onboarding eval fixture.
