# v1.1 Flow Redesign — Plan

**Status**: Draft for /autoplan review
**Source**: post-v1 user test feedback (8 items)
**Branch**: `soleh-ventures/v1.1-plan`

---

## Premise (from user feedback)

After live-testing v1, a runner experienced these gaps:

1. Strava connect happens late — the onboarding asks about training history that Strava already has
2. The "thinking…" ack arrives randomly and feels too clever; runners want a plain signal the moment they send a message
3. Post-onboarding has no anchor — MARP says "ask me anything" but a runner training for a race wants their plan
4. The plan, when MARP describes one, doesn't show its reasoning — trust is lower than it could be
5. Onboarding feels open-ended — no sense of how many questions remain, no explanation of why each question matters
6. MARP is purely reactive — never reaches out, even when a planned workout is tomorrow
7. Plan personalisation feels surface-level — MARP doesn't deeply gather constraints before proposing
8. Even when a workout time is known, the runner has to manually add it to their calendar

These are real product gaps, not preference quirks. v1.1 should close them as a coherent flow redesign, not eight isolated patches.

## Eight items, mapped to the system

### Item 1: Strava-first onboarding

**Today**: Privacy notice → consent → onboarding (6+ turns) → Strava offer at end.

**v1.1**: Privacy notice → consent → **Strava prompt FIRST** (like "log in with Google") → onboarding that skips questions Strava already answers.

What Strava actually gives us (via the 60-day backfill):
- Activities (distance, duration, HR, pace, dates)
- Strava athlete ID + scope
- Implicitly: weekly volume, long-run distance, training frequency, recent race attempts

What Strava does **not** give us:
- Name
- Goal race + date
- Injury history
- Life context (work, travel)
- Coaching preferences

So onboarding becomes shorter, not skipped. The question "how many days a week do you train?" can be answered from Strava data with the runner just confirming.

If runner skips Strava: full onboarding runs as it does today.

**Files**:
- `src/services/process-incoming.ts` — after consent, send the Strava prompt BEFORE the onboarding branch
- `src/services/strava-connect.ts` — add a "pre-onboarding" entry-point that's not an LLM intent detection (it's the explicit next step)
- `src/flows/onboarding.ts` — when athlete has a Strava connection, derive what we can; only ask what's missing
- `prompts/onboarding.md` — new prompt section: "if Strava context is provided, confirm rather than ask"

### Item 2: Simple "thinking…", fired on every routed message

**Today**: 8-phrase random ack, fires after 5 s.

**v1.1**: Single phrase "thinking…", fires immediately when the message starts being routed (not on consent / dormancy / file-ingest / Strava-connect / deletion paths — those are fast).

Trade-off acknowledged: the 8-phrase variety was a personality choice. Going to "thinking…" trades warmth for clarity. The user is explicit about wanting clarity; this is the right call.

Edge case: very-fast replies (cached classifier, single domain, ~1.5 s). The "thinking…" still ships, then 1.5 s later the real reply lands. Slight chat clutter but acceptable for honesty.

**Files**:
- `src/services/thinking-ack.ts` — collapse to one phrase, remove timer
- `src/webhooks/twilio.ts` — fire immediately (not via `setTimeout`) when processing enters the routing branch
- Move the firing point from the generic webhook to a narrower hook so it only triggers when the LLM router will actually run

### Item 3: Post-onboarding pivots to training plan

**Today**: onboarding ends with "ask me anything" → no anchor.

**v1.1**: onboarding ends with one specific question:

> Two ways to start: (a) you already have a training plan and want me to coach you through it, or (b) you'd like me to build one for you from scratch. Which one?

Both branches require new code:

- **(a) Bring your own plan** — runner pastes / describes their existing plan; MARP captures it as structured `race_blocks.plan` jsonb. New service: `src/services/plan-ingest.ts`.
- **(b) Build from scratch** — kicks off the pyramid flow (Item 7). New service: `src/services/plan-generator.ts` (the T8 periodised planner from `TODO.md`).

This expands v1.1 scope significantly. The training plan becomes a first-class artifact, not a deferred feature.

### Item 4: Scientific reasoning in plan output

**Today**: domain prompts (especially `training.md`) cite the principle in one phrase (ET18). Plans described conversationally rarely surface the why.

**v1.1**: When MARP presents a plan (or a plan change), every meaningful decision includes a one-sentence why. Examples:

- "Week 1: 38km total — building from your current 30 (10%-rule)"
- "Long run capped at 30km, 3 weeks out — Pfitz taper structure"
- "Tempo Wednesday at HR 165 — lactate-threshold work for marathon endurance"

Implementation is mostly prompt-side. The plan-generator prompt (Item 3) embeds this requirement; the training domain prompt is already partway there.

**Files**:
- `prompts/plan-generator.md` (new) — bakes the cite-the-reasoning rule into every weekly output
- `prompts/domains/training.md` — strengthen the existing rule to require reasoning on plan-shaped advice

### Item 5: Onboarding progress + per-question rationale

**Today**: onboarding is LLM-driven freeform. No progress signal. No per-question why.

**v1.1**: every onboarding reply ends with two lines:

```
Why I ask: [one line — what this lets me coach better]
Onboarding: ~3 of 6 (about 2 min left)
```

Approach: keep the LLM-driven onboarding (preserves conversational feel) but add **stages** the LLM knows about. Six stages, each emits its rationale + progress tail.

Stages (rough):
1. **Identity** — name, locale
2. **Goal** — race name, distance, date, goal time
3. **Fitness** — current weekly volume, recent long run (derive from Strava if connected; confirm rather than ask)
4. **History** — past races, prior injuries, years running
5. **Constraints** — current injuries, travel, life load, training-days-available
6. **Preferences** — accountability style, communication frequency, reminder opt-in (ties to Item 6)

Stage progress is computed in code; per-stage rationale is fixed copy in the onboarding flow (not LLM-generated, so it's consistent).

**Files**:
- `src/flows/onboarding.ts` — gain explicit stage tracking
- `prompts/onboarding.md` — instruct the LLM to ask one question per stage in sequence, defer to provided stage hints
- New: `src/flows/onboarding-stages.ts` with the rationale copy + progress math

### Item 6: Proactive daily reminders

**Today**: MARP is purely reactive.

**v1.1**: when a training plan exists with day-by-day schedule, MARP sends two messages per training day:

- **Night before, ~9pm runner-local time**: "Tomorrow: 12km easy at 6:00/km. Set the alarm, lay out the kit."
- **Morning of, ~5am runner-local time**: "Today's the 12km easy. Conditions look [X]. Ping me when you're done."

Configurable: runner can turn off, change times, switch to night-only or morning-only.

Infrastructure needed:
- Runner locale + timezone (extend `athletes` table; default from Twilio number's country if not stated)
- Reminder preferences (`athletes.reminder_prefs` jsonb: `{ enabled, night_before, morning_of, timezone }`)
- A scheduler: simplest viable = Railway cron hitting `/internal/cron/reminders` every 15 min; the endpoint queries which reminders are due in the current window.

**Files**:
- New migration: add `timezone` + `reminder_prefs` columns to athletes
- New: `src/routes/cron-reminders.ts` + `src/services/reminder-scheduler.ts`
- New: `src/services/reminder-templates.ts` with the night/morning copy
- Railway: add a cron schedule pointing at `/internal/cron/reminders` (with `CRON_SECRET` header auth)

### Item 7: Pyramid plan-building

**Today**: no plan generator at all.

**v1.1**: when runner picks "build from scratch" (Item 3b), MARP doesn't dump a 16-week plan. It works through three tiers with commitment checkpoints:

```
Tier 1: Big picture (5 min)
  ↓ Goal: race + date + goal time
  ↓ Constraints: days/week, weekly mileage ceiling, life events
  ↓ Output: macro shape ("16-week plan, peak week 60km, build → peak → taper")
  ↓ Commit: "Does this shape match your reality?"

Tier 2: Weekly cadence (3 min)
  ↓ Show week 1 + week 2 in detail
  ↓ Show key sessions per week (long run + 1 quality session)
  ↓ Output: weekly skeleton for the full block
  ↓ Commit: "Ready to commit to this rhythm?"

Tier 3: Daily detail (on-demand)
  ↓ Each day's session detailed when asked
  ↓ Or pushed via reminders (Item 6)
```

Each tier ends with an explicit commit checkpoint. This connects to v1's decision-frame binder — a "commit?" question IS a decision frame.

**Files**:
- New: `src/services/plan-generator.ts` (the T8 service)
- New: `prompts/plan-tier1-macro.md`, `prompts/plan-tier2-weekly.md` (one prompt per tier; tier 3 reuses the existing training domain)
- `src/db/schema.ts` — `race_blocks.plan` jsonb gets a real schema (weeks + sessions)
- Memory retrieval surfaces "this week's planned sessions" alongside recent activities

### Item 8: Calendar export

**Today**: runner manually translates "Wednesday tempo 6am" into a calendar entry.

**v1.1**: when a session has a known date + time, MARP appends a calendar link:

```
Wednesday tempo — 6 km @ HR 165
Add to calendar: https://marp.com/cal/<token>.ics
                 https://www.google.com/calendar/render?action=TEMPLATE&text=...
```

Two delivery paths:

1. **.ics file** — hosted at `/cal/<signed-token>.ics`, signed with `MAGIC_LINK_SECRET`, expires after the session. Tap → adds to Apple Calendar / Outlook / Google Calendar via standard ICS handling.
2. **Google Calendar quick-add URL** — the `https://www.google.com/calendar/render?action=TEMPLATE&...` format. Works for Google users in one tap; doesn't require hosting.

Ship both for v1.1 — the ICS file covers Apple/Outlook users, the Google URL is one-tap for the Google majority.

**Files**:
- New: `src/routes/cal.ts` — `GET /cal/:token.ics` that verifies the token and renders an iCalendar response
- New: `src/services/calendar-export.ts` — builds the ICS body + the Google quick-add URL
- Reminder templates (Item 6) include the link
- Plan-tier output (Item 7) includes the link

## Sequencing

Items map to two parallel lanes:

**Lane A — UX polish (small, independent)**
- Item 2 (simple thinking…) → ~30 min
- Item 5 (progress + rationale) → ~3 hr
- Item 1 (Strava-first restructure) → ~2 hr

**Lane B — Training plan as a feature (big, sequenced)**
- Item 3 (post-onboarding pivot) → ~1 hr (just the branching; the actual generator is Item 7)
- Item 7 (pyramid plan-build) → ~6–8 hr — the centrepiece
- Item 4 (scientific reasoning) → ~1 hr (mostly prompt work, but coupled to Item 7's output)
- Item 6 (proactive reminders) → ~4 hr (cron infra + preferences + templates)
- Item 8 (calendar export) → ~2 hr

**Total estimate**: ~21 hours of focused build work. Realistic timeline 1–2 weeks.

Recommended order:
1. Lane A in parallel (small UX wins, ship continuously)
2. Lane B in sequence: Item 3 (branching scaffold) → Item 7 (planner) → Item 4 (reasoning in plan output) → Item 6 + Item 8 (reminders + calendar can ship together once plans exist)

## What this plan does NOT change

- Privacy / consent flow (v1 stays correct)
- Strava webhook subscription_id guard (v1 stays correct)
- Memory layer (extends; the new `plan` jsonb populates the existing `race_blocks.plan` column)
- Erasure / dormancy (untouched)
- Decision binder (the pyramid commit checkpoints REUSE the binder — no new mechanism)
- LLM router (the existing classifier / domains / synthesizer still handle freeform questions; the plan generator is a parallel pipeline triggered by an explicit intent, not a router-routed call)

## Open questions

1. **Timezone capture**: do we ask in onboarding, infer from country code, or both? Inferring is faster but error-prone for travellers. Asking is reliable but adds a question.
2. **Plan format on import (Item 3a)**: pasted text? URL to a TrainingPeaks plan? PDF upload? v1.1 probably starts with pasted text only.
3. **Reminder timing customisation**: do we accept any time string in onboarding, or fixed presets (5am/9pm only, with adjust later)?
4. **Plan revision**: if a runner gets a plan and life changes, how do they ask for a re-cut? Implicit (chat → MARP infers → regenerate) or explicit ("/replan")?

### 0A. Premise challenge

The plan rests on three implicit premises. Two hold; one is worth surfacing.

| # | Premise | Verdict |
|---|---|---|
| P1 | "Runners training for a race want a plan" | **Holds.** The user's feedback (Item 3, Item 7) is the loudest signal possible. Conversational coaching without an artifact is what they explicitly rejected. |
| P2 | "Proactive reminders increase engagement, not annoyance" | **Holds, conditionally.** Opt-in framing keeps it from being intrusive. The risk is reminder fatigue — many runners install habit apps and uninstall them within a month for exactly this reason. ### 0C. Implementation alternatives

The plan ships all 8 items as v1.1. Two real alternatives:

**Alt A (recommended sequencing):** Split v1.1 into v1.1a + v1.1b.
- v1.1a: Lane A only (Items 1, 2, 5) — UX polish, ~5–6 hr, ship this week, validate with the same beta runner
- v1.1b: Lane B (Items 3, 4, 6, 7, 8) — training plan as a feature, 2–3 weeks, commit AFTER v1.1a feedback confirms direction
- Why: lets the planner work be informed by what actually moves the v1.1a metrics. If Strava-first + progress indicators alone hit the onboarding-completion target, the planner can be designed differently than if they don't.

**Alt B (plan-as-MVP):** Ship Item 3 + Item 7 with a STUB planner that emits a 3-week plan only (not full periodisation), validate, then build the full pyramid.
- Why: gets the artifact in front of runners faster, with lower hallucination surface
- Risk: a 3-week plan for a 16-week race is barely useful

**Alt C (ship-all-at-once as proposed):**
- Why: the items are coherent — items 4/6/8 don't make sense without item 7's plan
- Risk: 3+ week build with no intermediate user signal

**Recommendation**: Alt A. Splits the bet without losing coherence. Lane A still ships items that improve every runner's experience, Strava-connected or not.

- Reminder cron: each fire is <500 ms (templated, no LLM). At 100 active runners, 200 reminders/day = trivial.
- ICS endpoint: <100 ms (just template + HMAC verify).

Database:
- Reminder cron query: index on `athletes.reminder_prefs->>'enabled' = 'true'` + `race_blocks WHERE state='active'`. Plan adds appropriate indices.
- Plan storage: race_blocks.plan jsonb fits comfortably; weekly plans are < 8KB each.

### Section 5: Security

| Surface | Risk | Mitigation |

| Reminder cron timezone | Wrong-time reminder | Fail-safe: if timezone is missing, skip reminder entirely; never guess |
